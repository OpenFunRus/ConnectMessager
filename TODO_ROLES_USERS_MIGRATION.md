## 2) Что уже реализовано в `test`

Источник: `test/index.html`, `test/app.js`, `test/styles.css`.

### 2.1 Основная модалка "Настройки"

- Открытие по кнопке шестеренки (вверху и в карточке пользователя).
- Заголовок: `Настройки`.
- Два таба на всю ширину:
  - `Роли`
  - `Пользователи`
- Табы переключают два независимых списка.
- Видимость кнопки шестеренки: только для роли с правом `Видеть настройки`.

### 2.2 Вкладка "Роли"

- Отображается список ролей.
- У роли:
  - название + цветной маркер;
  - подсказочный текст/сводка;
  - справа блок из 3 кнопок `50x50`:
    - `Ранг` (число, фон цвет роли),
    - `Редактировать` (Tabler `pencil.svg`),
    - `Код доступа` (Tabler `key.svg`).
- Кнопка `Добавить роль` под списком.
- Видимость кнопок:
  - `Добавить роль` — право `Добавлять роли`
  - `Редактировать` — право `Редактировать роли`
  - `Код доступа для роли` — право `Генерировать код доступа для роли`

### 2.3 Вкладка "Пользователи"

- Есть поиск по имени (фильтрация в реальном времени).
- Отображается список пользователей.
- У пользователя справа 4 кнопки `50x50`:
  - `Ранг`,
  - `БАН` (Tabler `ban.svg`),
  - `Редактировать` (Tabler `pencil.svg`),
  - `Код входа` (Tabler `key.svg`).
- Видимость кнопки `Код входа пользователя` — право `Генерировать код входа для пользователей`.
- `Редактировать` пользователя: отдельная модалка, пока меняется только имя.

### 2.4 Модалка "Добавить/Редактировать роль"

- Заголовок:
  - при создании: `Добавить роль`,
  - при редактировании: `Редактировать роль`.
- Поля (построчно):
  - Название роли
  - Цвет роли (выбор из пула 32 цветов, выпадающий список)
  - Ранг роли
  - Фильтр роли (Глобальный / Фильтр)
  - Фильтр (включается/выключается в зависимости от выбора)
- Блок прав:
  - Видеть настройки
  - Добавлять роли
  - Редактировать роли
  - Генерировать пригласительный код группы
  - Генерировать пригласительный код пользователей
- Блок ограничений:
  - сообщения в минуту
  - запросы к серверу
  - символы в сообщении
  - строки в сообщении
  - размер файлов
  - количество файлов в сообщении (по умолчанию 9)
  - форматы файлов
- Блок возможностей:
  - звонки
  - видеовызовы
  - просмотр удаленного рабочего стола

### 2.5 Модалка "Код доступа"

- Заголовок: `Код доступа`.
- Код из 8 цифр (визуально отдельными плитками).
- Кнопки:
  - Обновить (Tabler `refresh.svg`)
  - Копировать (Tabler `copy.svg`)
- Есть два разных сценария кода:
  - код роли:
    - при генерации сразу создается временный пользователь на сервере с нужной ролью;
    - вход по коду открывает именно эту предсозданную учетную запись;
    - если имя не назначено в течение 15 минут, временный пользователь должен быть удален;
  - код пользователя:
    - выдается на существующую учетную запись;
    - работает как одноразовый динамический пароль для входа;
    - после успешного входа код должен сгорать.

### 2.6 Модалка подтверждения удаления пользователя (БАН)

- Текст подтверждения:
  - удаление пользователя,
  - удаление всей истории,
  - удаление медиафайлов,
  - удаление упоминаний.
- Кнопки:
  - Отмена
  - Удалить
- Семантика `БАН` в текущем ТЗ: это полное удаление пользователя (не мягкий бан).

### 2.7 Модалка редактирования пользователя

- Пока только изменение имени пользователя.
- Кнопки:
  - Отмена
  - Сохранить

### 2.8 Ограничения и проверки на прикреплении/чате

- Ограничения символов/строк влияют на чатовый ввод и должны отображаться в UI ввода как текущие лимиты роли.
- Ограничение количества файлов в сообщении применяется в модалке прикрепления:
  - по достижении лимита нельзя добавлять новые файлы (отключать/скрывать `+`).
- Формат и размер файлов проверяются на клиенте до загрузки на сервер (pre-check), и дублируются серверной валидацией.
- Ограничения запросов/сообщений требуют серверной обвязки и централизованного управления лимитами роли.

### 2.9 Города и изоляция по фильтру

- `Фильтр городов` — критический параметр.
- Пользователь с фильтром города (например, Самара):
  - видит только пользователей/группы Самары;
  - не может писать в Москву и взаимодействовать с чужим городом.
- Пользователь с глобальной ролью:
  - видит всех пользователей и все группы.

## 4.2 Backend (`apps/server`)

Нужно определить боевую модель (миграции и API):

### A. Роли

- CRUD ролей.
- Права роли (5 новых флагов).
- Ограничения роли (с числами/списком форматов).
- Возможности роли (звонок/видео/remote-desktop).

### B. Пользователи

- Получение списка пользователей.
- Поиск по имени.
- Обновление имени.
- Удаление пользователя (каскад):
  - сообщения,
  - медиафайлы,
  - упоминания/ссылки,
  - закрепы (если message удален),
  - ЛС с другими пользователями (включая чистку read-state/unread-counters).
- После удаления не должно оставаться "зависших непрочитанных" в интерфейсе.

### C. Коды доступа и входа

- Два отдельных production-сценария:
  - `role_access_code`:
    - при генерации сервер сразу создает временного пользователя;
    - временный пользователь сразу получает выбранную роль;
    - код ведет в уже созданную учетную запись, а не создает ее в момент логина;
    - если имя не задано в течение 15 минут, временный пользователь удаляется cleanup-job'ом;
  - `user_login_code`:
    - выдается на существующую учетную запись;
    - используется как одноразовый динамический пароль для входа;
    - после успешного входа код сгорает.
- Генерация кода: 8 цифр.
- Срок жизни кода:
  - действует до 15 минут **или** до первого использования;
  - после использования или истечения TTL код уничтожается на сервере.
- Код должен храниться на сервере только в hash-виде.
- Для очистки:
  - удаление сразу при successful use;
  - cron-очистка просроченных временных пользователей и просроченных кодов.

### D. Аудит и логирование (подробные логи)

Нужно добавить подробный аудит по критичным действиям:

- генерация/использование/истечение/удаление пригласительных кодов;
- кто и когда:
  - добавил роль,
  - отредактировал роль,
  - изменил права/лимиты роли;
- кто и когда редактировал пользователя;
- кто и когда удалил пользователя (с указанием инициатора и цели);
- события входа:
  - успешный/неуспешный логин,
  - вход по user_relogin_code,
  - смена IP пользователя,
  - вход в группу по group_invite_code.

Рекомендуемые поля аудита:

- `eventType`
- `actorUserId`
- `targetUserId` / `targetRoleId` / `targetInviteId`
- `sourceIp` / `newIp` / `oldIp`
- `payload` (json: старые/новые значения)
- `createdAt`
- `result` (`success` / `failed`) + причина ошибки (если есть)

**Что проверить вам:**

- `Генерировать пригласительный код группы` влияет только на кнопку кода группы;

### Итерация 11.7 (выполнено, перенос policy роли на сервер)

**Проблема:**

**Сделано:**

- Расширена таблица `roles`:
  - `rank`
  - `scope`
  - `filter`
  - `limits`
  - `abilities`

- Роуты `roles.add` / `roles.update` теперь создают и сохраняют полный policy snapshot роли на сервере.
- Клиентский `prototype-interface` переведен на чтение расширенных server roles:

  - лимиты ввода/файлов;
  - способности;
  - списки ролей в настройках.
- Для живого применения без ручного refresh использованы текущие role subscriptions (`ROLE_UPDATE` / `ROLE_CREATE` / `ROLE_DELETE`), которые теперь несут расширенную роль.
- Серверные проверки переведены на policy роли для критичных сценариев:
  - `messages.send`: символы, строки, количество файлов;
  - upload HTTP: формат файла, размер файла.

**Важно:**

- Это уже реальный server-side источник для основных role settings.
- При изменении роли значения приходят всем клиентам через существующие subscriptions.
- Полный server-side охват еще не завершен для:
  - изоляции по городам (`scope/filter`) на уровне выборок/доступов;
  - принудительного закрытия/сброса запрещенных экранов при потере прав.

**Что проверить вам:**

- меняете лимиты роли -> у пользователей этой роли сразу меняются ограничения в UI;
- меняете `rank/scope/filter/abilities` -> клиент подхватывает новые значения без ручного пересоздания роли;
- если роль ограничивает формат/размер файла, сервер больше не пропускает такой файл.

- При выборе роли по инвайту новый пользователь больше не получает поверх нее дефолтную роль `Member`, чтобы не терялся фильтр приглашенной роли.
- При равном ранге приоритет основной роли отдан фильтруемой роли, а не глобальной.

**Что проверить вам:**

- новый пользователь появляется у других без ручного refresh;
- разработчик и пользователь видят друг друга, если фильтры совместимы;
- если фильтр становится несовместимым, контакт исчезает и открытый чат закрывается;
- инвайт в роль с фильтром действительно выдает пользователю этот фильтр, а не `Все`.

**Что проверить вам:**

### Итерация 11.15 (выполнено, requestsPerMinute больше не ломает обычный чат)

**Проблема:**

- Ролевой лимит `requestsPerMinute` применялся через общий `protectedProcedure`, то есть буквально ко всем защищенным tRPC-вызовам.
- Для роли 10 это ломало обычный lifecycle клиента: загрузка истории, синхронизация DM, подписки и typing-сигналы быстро выжигали лимит.
- Симптомы совпадали с поведением пользователя `Кирова 304а`: сначала ничего не грузится/не отправляется, потом через минуту окно лимитера сбрасывается и чат временно “оживает”, после чего снова умирает.

**Сделано:**

- В `apps/server/src/utils/trpc.ts` лимит `requestsPerMinute` больше не расходуется на `query` и `subscription`.
- Отдельно исключен `messages.signalTyping`, чтобы индикатор набора не блокировал отправку сообщений у низких ролей.
- Ролевой лимит запросов теперь остается только на значимых `mutation`, а не на пассивном realtime/lifecycle чата.

**Что проверить вам:**

- у пользователя уровня 10 история и DM загружаются сразу, без “оживания через минуту”;
- сообщения стабильно отправляются подряд, а не только после ожидания окна лимитера;
- индикатор набора текста не приводит к тому, что дальнейшая отправка внезапно перестает работать.

  - выдает одноразовый код доступа;
  - пользователь при первом входе задает имя;
  - если имя не задано 15 минут, временный пользователь удаляется.
- Кнопка на пользователе:
  - выдает одноразовый код входа в существующую учетную запись;
  - код одноразовый и хранится в hash-виде.

**Что проверить вам:**

- генерация кода на роль 10 больше не ведет в разработчика 100;
- генерация кода на пользователя логинит именно в выбранную учетку;
- после успешного входа повторное использование того же кода не работает;
- временный пользователь без имени должен исчезать после TTL.

### Итерация 11.40 (выполнено, handoff-fix миграционного journal для production exe)

**Сделано:**

- После запуска нового `exe` был найден production-handoff баг:
  - сервер видел `0011/0012/0013.sql` в live `drizzle`,
  - но не применял их, потому что `apps/server/src/db/migrations/meta/_journal.json` не содержал соответствующих записей.
- В `meta/_journal.json` добавлены записи:
  - `0011_grant_pin_messages_to_default_roles`
  - `0012_backfill_default_role_for_users_without_roles`
  - `0013_add_user_login_codes_and_pending_activations`
- После этого `build-exe.bat` пересобран повторно, чтобы новый `drizzle.zip` попал внутрь production `exe`.

**Зачем это сделано:**

- Без этого другой компьютер или свежий production-launch могли бы стартовать с live SQL-файлами, но без применения новых миграций.
- Это критичный пункт для handoff между машинами.

**Что помнить при продолжении на другом ПК:**

- брать только свежий `connectmessager-server.exe`;
