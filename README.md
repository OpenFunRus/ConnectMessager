# Connect Messager

Connect Messager - корпоративный мессенджер для внутренней коммуникации команд.

## Текущий статус

- В корень проекта перенесена рабочая кодовая база (apps/packages) для дальнейшей разработки.
- Проведен ребрендинг под Connect Messager (нейминг проекта, сборка, переменные окружения).
- Добавлен UI-прототип мессенджера в `test`.
- Добавлена отдельная страница входа `test/login.html`.
- Настроена сборка Windows `exe` и батник `build-exe.bat`.
- Реализована серверная логика входа:
  - первый пользователь создается автоматически как "Разработчик" по паролю `12345678`;
  - разработчик получает только роль владельца (без `Member`);
  - для role access flow создается временный пользователь с авто-`identity` в формате `cm-user-xxxxxxxxxxxx`;
  - для user access flow выдаются одноразовые коды входа в существующую учетную запись.
- Реализован автологин после разрыва соединения с бесконечными попытками восстановления:
  - первый реконнект через случайный интервал `1-60 секунд`;
  - все следующие попытки каждые `60 секунд`;
  - на экране ожидания показываются кнопка `Обновить страницу` и countdown до следующей попытки.
- Добавлена password-first логика: если введенный пароль совпадает с существующим пользователем, выполняется вход в существующий аккаунт без генерации нового `cm-user-id`.
- Добавлены серверные `info`-логи авторизации (`/login`, `/login/restore`) с причинами отказа и деталями сравнения IP.
- Добавлена новая access-code модель:
  - `users.provisionRoleAccess` — временный пользователь под роль + одноразовый код доступа;
  - `users.issueLoginCode` — одноразовый код входа для существующего пользователя;
  - `cleanup-pending-users` — удаление неактивированных временных пользователей по TTL.
- Из начального seed убраны дефолтные пользователь, каналы и сообщения.
- Добавлено полное удаление пользователя с каскадной очисткой:
  - сообщения;
  - упоминания;
  - закрепы;
  - реакции;
  - GIF и вложения;
  - сама учетная запись.
- Добавлены управляемые группы с фильтрами видимости:
  - отдельное право `MANAGE_GROUP_CHATS`;
  - создание/редактирование/удаление групп;
  - аватарка и описание;
  - realtime-скрытие и показ группы при смене фильтра.
- Добавлены упоминания и quote-flow:
  - `@`-autocomplete;
  - уведомления в верхнем колокольчике;
  - цитирование с переходом к оригиналу;
  - цитирование текста и GIF.
- Добавлена кнопка выхода с confirm-модалкой и полной очисткой локального автологина.
- Починен typing-индикатор для групп и множественного набора.
- Emoji picker переведен на категорийные кнопки-иконки `8` в ряд.
- Добавлены реакции на сообщения с picker-попапом, переключением по повторному клику и кнопкой `+`.
- Реакции разрешены всем пользователям, без отдельного permission-гейта.
- Добавлены звуки отправки и получения сообщений в production-прототипе.
- В редактирование собственного профиля добавлена смена аватарки.
- Аватарки теперь корректно отображаются внизу слева, в списке контактов и у сообщений в чате.

## Linux / Ubuntu production

В репозитории уже есть production-путь для Linux через Docker:

- `Dockerfile`
- `docker-entrypoint.sh`
- `docker-compose.yml`

По умолчанию используются:

- web-порт: `4991/tcp`
- WebRTC TCP: `40000/tcp`
- WebRTC UDP: `40000/udp`
- data path внутри контейнера: `/home/bun/.config/connectmessenger`
- data volume на хосте: `./data`

### Удаление старого OpenCord

Если на сервере уже крутится старая установка OpenCord, сначала лучше корректно остановить и убрать её, а данные сохранить в backup:

```bash
cd ~/opencord || cd /opt/opencord
docker compose down
cp -r data data-backup-$(date +%F-%H%M%S)
cd ..
mv opencord opencord-old-$(date +%F-%H%M%S)
```

Если OpenCord был установлен не через Docker Compose или лежит в другом пути, замени путь на свой фактический каталог.

### Установка Connect Messager на Ubuntu

```bash
sudo apt update
sudo apt install -y docker.io docker-compose-v2 git
sudo systemctl enable --now docker

cd ~
git clone https://github.com/OpenFunRus/ConnectMessager.git
cd ConnectMessager
```

Перед запуском открой `docker-compose.yml` и укажи реальный публичный IP или домен в:

- `CONNECTMESSAGER_WEBRTC_ANNOUNCED_ADDRESS`

Если используется `ufw`, открой порты:

```bash
sudo ufw allow 4991/tcp
sudo ufw allow 40000/tcp
sudo ufw allow 40000/udp
```

Запуск:

```bash
docker compose up -d --build
docker compose ps
docker compose logs -f --tail=200
```

После старта интерфейс будет доступен по адресу:

- `http://SERVER_IP:4991`

### Обновление Linux-установки

```bash
cd ~/ConnectMessager
git pull origin main
docker compose up -d --build
docker compose ps
docker compose logs -f --tail=200
```

### Примечания по миграции с OpenCord

- Старый OpenCord использовал свои env-переменные `OPENCORD_*`; у Connect Messager нужны `CONNECTMESSAGER_*`.
- Старый production data path был завязан на `opencord`, новый путь внутри контейнера: `/home/bun/.config/connectmessenger`.
- Если нужно сохранить старую историю/базу перед переходом, сначала сделай backup каталога `data`.
- Контейнер в `docker-compose.yml` уже настроен с `restart: unless-stopped`.

## Авторизация

- Экран входа в `apps/client` оформлен в стиле `test/login.html`.
- Поле ввода кода скрыто (`password`) и имеет кнопку "глаз" для показа/скрытия.
- Поддерживается единое поле: "Введите код доступа или пароль".
- При повторном входе используется сохраненный `CM_USER_ID` (если валиден), чтобы identity не "прыгал" между логинами.
- Текст ошибки входа: `Неверный код доступа или пароль`.
- Код для роли:
  - создает временного пользователя с заданной ролью;
  - требует задать имя при первом входе;
  - если имя не задано в течение `15 минут`, такой пользователь удаляется сервером.
- Код для пользователя:
  - выдается на конкретную существующую учетную запись;
  - одноразовый;
  - хранится на сервере только в хеше и сгорает после успешного входа.

## Автовосстановление соединения

- После успешного входа сохраняются `CM_USER_ID` и `CM_USER_IP` в `localStorage`.
- При обрыве соединения клиент остается на экране автоподключения.
- Первый авто-реконнект выполняется через случайный интервал от `1` до `60` секунд.
- Все следующие попытки восстановления идут раз в `60` секунд.
- На экране автоподключения показываются кнопка `Обновить страницу` и таймер до следующей попытки.
- Для `KICKED`/`BANNED` автологин не выполняется.
- `login/restore` проверяет IP с приоритетом серверного последнего IP пользователя; при локальном IP (`192.168.x.x`) выполняется сопоставление внешнего IP для корректной работы в LAN.

## Обновление фронта после сборки (без Ctrl+F5)

- Добавлен маркер сборки `buildId` в ответ `GET /info`.
- Клиент сравнивает `buildId` с сохраненным значением и при изменении делает однократный `window.location.reload()`.
- Защита от циклических перезагрузок: для одного и того же `buildId` reload выполняется не более одного раза за сессию.
- Это закрывает кейс локальной разработки "страница открыта, сервер перезапущен, был только автореконнект без полного рефреша".

## Актуальный UI в apps/client (server-view)

- Новый интерфейс из `test` перенесен в `apps/client/src/screens/server-view/prototype-interface.tsx` + `prototype-interface.css`.
- Реализована полноценная DM-модель для "Заметки" и контактов через TRPC (`dms.open`, `messages.get`, `messages.send`), сообщения сохраняются в БД.
- Лента сообщений поддерживает поведение в стиле OpenCord:
  - окно `50` последних сообщений;
  - расширение до `100`;
  - догрузка вверх/вниз чанками;
  - работа через before/after буферы;
  - возврат к актуальному хвосту ленты.
- Добавлена оверлей-кнопка прокрутки вниз в чате (Tabler-стиль), с автопоявлением при смещении от низа.
- Сообщения группируются по автору и времени (как в Discord/OpenCord): повторяющиеся быстрые сообщения не дублируют аватар/ник.
- Поддержка `Enter` для отправки и `Shift+Enter` для новой строки в поле ввода.
- Относительное время сообщений выводится через `RelativeTime`.
- В тексте сообщений сохранены переносы строк (многострочный ввод корректно отображается в чате).
- Под сообщениями отображаются реакции:
  - свои реакции мягко подсвечиваются синим;
  - счетчик показывается только от `2`;
  - через отдельную кнопку `+` можно добавить другую реакцию.
- Индикатор `печатает...` работает:
  - в `Заметках`;
  - в DM;
  - в общей группе;
  - в кастомных группах;
  - для нескольких пользователей одновременно.
- Для production-прототипа подключены message sounds:
  - `MESSAGE_SENT` на текст/GIF/вложения;
  - `MESSAGE_RECEIVED` на входящее realtime-сообщение не от себя.

## Эмодзи (Twemoji only, без системных emoji в отображении)

- Emoji picker в композере:
  - блок "Последние" (до 16, 2 ряда по 8);
  - категории как квадратные кнопки-иконки `8` в ряд;
  - в каждый момент открыта только одна выбранная категория;
  - локальные Tabler/Twemoji ассеты;
  - сохранение последних в `localStorage`.
- Категории в picker локализованы на русский.
- Визуальный рендер сообщений использует Twemoji-картинки.
- Копирование emoji из чата и поля ввода стабилизировано через token-based подход (`:emoji_name:`), чтобы не было распада составных emoji.
- Размер emoji в сообщениях:
  - 1 emoji-only: `48px`
  - 2-5 emoji-only: `32px`
  - >5 emoji-only: `24px`
  - emoji + любой текст: `24px`

## GIF (текущий статус)

- В пикере добавлена вкладка `GIF` с поиском и выдачей до `10` элементов.
- Во время загрузки GIF показываются skeleton-плейсхолдеры, чтобы попап не "прыгал" по размеру.
- Клик по GIF закрывает попап и отправляет сообщение в чат.
- Сообщение хранится как URL с префиксом `cmx-gif-url:` (без загрузки файла GIF на сервер).
- В ленте GIF рендерится с ограничением `max-height: 320px`; вертикальные GIF сохраняют вертикальную ориентацию.
- Последние использованные GIF (до `10`) сохраняются локально и подмешиваются в верх выдачи без поиска.

## Группы

- Помимо общей группы поддерживаются отдельные управляемые группы на базе обычных `channels`.
- У группы есть:
  - имя;
  - короткое описание;
  - аватарка;
  - фильтр видимости.
- Видимость группы:
  - фильтр `Все` видят все;
  - фильтруемую группу видит только такой же фильтр и роль `Все`.
- Управление группами:
  - если у пользователя роль `Все`, он может редактировать все группы;
  - если у пользователя локальный фильтр, он может редактировать только группы своего фильтра;
  - клиент и сервер проверяют это одинаково.
- При смене фильтра группы список пользователей обновляется в realtime без перезахода.

## Удаление пользователя

- Удаление из настроек делает именно полную очистку, а не мягкий бан.
- После удаления:
  - пользователь не должен переавторизоваться;
  - его сообщения и связанные объекты исчезают и из групп, и из DM;
  - тексты UI локализованы на русский.

## Что важно учитывать сейчас

- Для production-проверки на другом ПК нужен свежий `connectmessager-server.exe` с упакованной миграцией `0014_group_channels`.
- Если после очистки `%appdata%` сервер падает на `is_group_channel`, значит запущена старая сборка без новой `drizzle/meta/_journal.json`.
- Базовая архитектура чата и хранения сообщений стабильна; текущая зона риска в основном в UX-polish и smoke-проверках на чистой Windows-машине.
