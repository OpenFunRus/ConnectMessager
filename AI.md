## Что уже реализовано в прототипе

- Собран тёмный интерфейс в корпоративном стиле (цвета/контраст/панели).
- Верхние 4 таба: "Контакты", "Группы", "Боты", "Другое".
- Табы сделаны адаптивной сеткой 4x1: все кнопки одинакового размера.
- Табы иконками без текста, с tooltip/aria-label для доступности.
- Между кнопками и отступы по краям блока табов: ровно `2px`.
- Скругления у кнопок табов уменьшены (небольшие, аккуратные).
- Кнопки табов растянуты по всей высоте верхнего меню (без пустоты снизу).
- Иконки используются только из локально скачанных Tabler SVG в `test/icons/tabler` (без inline-path).
- Поиски (левый и правый) без иконок, на всю ширину блоков, с отступом `6px` и небольшим скруглением.
- Работает поиск по контактам/группам и поиск сообщений в текущем чате.
- Работает выбор чата, плейсхолдер при невыбранном чате, локальная отправка сообщений.
- Список контактов/чатов: круглые аватарки с авто-цветом по имени (детерминированно), рядом ник/имя/название группы.
- Список контактов/чатов: расстояние `5px` между карточками и от краёв блока, внутренний отступ карточки `2px`.
- Блок текущего пользователя оформлен как отдельная карточка; справа в конце карточки кнопка настроек (Tabler settings), с hover/active анимацией.
- Текущая сетка размеров: верхний ряд `84px`, ряд поиска `50px`, нижний ряд (пользователь + ввод) `84px`.
- Левая колонка `330px`; верхние 4 кнопки по `80x80`.
- В карточках списка есть бейдж непрочитанных справа; по вертикали центрируется по всей карточке.
- На табах "Контакты/Группы" показывается синяя точка справа-снизу при наличии непрочитанных в разделе.
- Блок ввода сообщений: сначала поле ввода (по размерам/скруглению как поиск), затем 3 квадратные кнопки-иконки (смайл, файл, отправка) без текста.
- Кнопка смайлика открывает пикер с двумя вкладками: "Смайлики" и "GIF".
- Вкладка "Смайлики": "Последние" + общий пул (категории) с явным визуальным разделителем.
- Вкладка "GIF": поиск и выдача до `10` элементов, skeleton-плейсхолдеры во время загрузки.
- Ввод сообщений реализован с локальным раскрытием: стрелка появляется над полем (по центру) и работает без изменения общей сетки интерфейса.
- Раскрытие поля ввода выполняется как локальный оверлей внутри блока ввода (соседние блоки не двигаются), повторное нажатие сворачивает обратно.
- Блок ввода сообщений и его текущая стилизация зафиксированы как "идеальные" для текущего этапа.
- В области чата: при невыбранном чате отображается только текст "Добро пожаловать в Мессенджер Коннект".
- В заголовке активного чата показывается только имя/название чата (без строки статуса).
- В заголовке чата справа: кнопки звонка, видеозвонка и демонстрации экрана (пока disabled) + кнопка закрепов.
- Высота полосы заголовка чата `60px`; кнопки вызовов/демо/закрепов `50x50`.
- В шапке чата справа есть кнопка закрепов; по нажатию открывается панель закрепленных сообщений.
- В панели закрепов у каждого сообщения есть кнопка перехода к исходному сообщению в чате.
- После перехода панель закрепов закрывается, сообщение в ленте прокручивается в видимую область и подсвечивается пульсацией 2-3 раза.

## Текущая логика входа и кодов доступа (актуально)

- UI формы входа в `apps/client` сделан в стиле `test/login.html`.
- Единое поле: "Введите код доступа или пароль".
- Поле ввода скрыто (password) + кнопка "глаз" справа для показа/скрытия.
- На первом запуске (когда пользователей нет) вход выполняется по паролю `12345678`.
- Первый пользователь создается автоматически с именем `Разработчик`.
- Первый пользователь получает только роль владельца (без `Member`).
- Текущая целевая модель больше не должна опираться на старые invite-коды в settings UI:
  - кнопка на роли создает предзаведённого временного пользователя с нужной ролью;
  - этому пользователю выдается одноразовый код доступа на `15 минут`;
  - при первом входе по коду пользователь обязан задать имя;
  - если имя не задано до истечения TTL, временный пользователь удаляется cron-задачей.
- Кнопка на пользователе выдает одноразовый код входа для существующей учетной записи:
  - код хранится только в хеше;
  - после успешного входа код сразу сгорает;
  - это логический динамический пароль для входа в конкретную учётку.
- Для новых временных регистраций `identity` генерируется автоматически как `cm-user-xxxxxxxxxxxx` (12 символов, lower alnum).
- При повторном ручном входе клиент использует сохраненный `CM_USER_ID` из `localStorage` (если валиден) и не генерирует новый `identity`.
- Обычный пароль и password-first логика на сервере сохранены для классического входа без кода доступа.
- Сообщение об ошибке входа унифицировано: `Неверный код доступа или пароль`.
- Новые таблицы:
  - `user_login_codes`
  - `pending_user_activations`
- Новые серверные entry points:
  - `users.provisionRoleAccess`
  - `users.issueLoginCode`
- Новый cleanup:
  - `apps/server/src/crons/cleanup-pending-users.ts`
  - cron запускается раз в минуту.
- Важное ограничение на сейчас:
  - stateless JWT-сессии пока не отзываются автоматически после использования одноразового кода; если потребуется жесткая single-session модель, это следующий отдельный шаг.

## Автовход и реконнект (актуально)

- После успешного входа клиент сохраняет `CM_USER_ID` и `CM_USER_IP` в `localStorage`.
- При разрыве соединения клиент переходит в режим бесконечного автовосстановления, без экрана "Соединение потеряно".
- Первый реконнект запускается через случайный интервал `1-60 секунд`, чтобы все клиенты не ломились одновременно.
- Все следующие попытки восстановления идут каждые `60 секунд`.
- На экране `Автоматический вход... / Ожидание подключения к серверу...` теперь есть:
  - кнопка `Обновить страницу`;
  - countdown до следующей попытки.
- Автовосстановление отключено только для причин отключения `KICKED` и `BANNED`.
- `POST /login/restore` выполняет проверку `identity` и IP:
  - основной источник правды - последний серверный IP пользователя (как в модерации/логах входа),
  - fallback - `savedIp` из клиента, если серверный логин-IP отсутствует.
- Добавлена нормализация IP перед сравнением (IPv4/IPv6 mapped).
- Для локальных адресов (например `192.168.x.x`) добавлено сопоставление с внешним IP через `ipinfo`, чтобы корректно работать в LAN-сценарии.
- При `ip-mismatch` очищаются данные автовхода и показывается форма входа.
- При временной недоступности сервера автовход не сбрасывается и продолжает ретраи.

## Диагностика авторизации

- В серверной консоли добавлены `info`-логи по `login` и `login/restore`:
  - попытка входа, успешный вход, `identity`, `userId`, `requestIp`,
  - сравниваемые IP (`requestIp`, `lastKnownIp`, `savedIp`),
  - причина отказа (`missing-request-ip`, `last-login-ip-mismatch`, `saved-ip-mismatch`),
  - результат `resolvedPublicIp` для локальных IP.
- Эти логи включены в обычном режиме сервера (без необходимости debug-режима).

## Чанк-лента сообщений (актуально)

- В `prototype-interface` реализована оконная модель сообщений:
  - текущий хвост `50` сообщений;
  - расширение до `100`;
  - догрузка истории вверх;
  - догрузка новых вниз, если пользователь ушел в историю;
  - before/after буферы по аналогии с OpenCord-подходом.
- Вход в чат:
  - если есть непрочитанные - прыжок к первому непрочитанному;
  - иначе - в низ чата.
- После отправки собственного сообщения выполняется прокрутка в низ.
- Есть плавающая кнопка "вниз" (оверлей над чатом) с прокруткой к актуальному хвосту.

## Визуальное поведение сообщений (актуально)

- Сообщения группируются по автору и времени (близко к Discord/OpenCord):
  - первый элемент группы показывает аватар/ник/время;
  - последующие сообщения той же быстрой серии идут компактно.
- Включен мягкий hover-эффект строки сообщения.
- Переносы строк из ввода корректно сохраняются и отображаются в ленте.
- Реальные аватарки теперь протянуты:
  - в списке контактов;
  - внизу слева в карточке текущего пользователя;
  - у сообщений в самой ленте.
- В prototype подключены звуки сообщений:
  - `MESSAGE_SENT` на текст, GIF и вложения;
  - `MESSAGE_RECEIVED` на входящее realtime-сообщение не от себя.

## Emoji система (Twemoji only, актуально)

- Emoji picker:
  - "Последние" сверху, 2 ряда по 8;
  - дальше категории кнопками-иконками, `8` в ряд;
  - одновременно показывается только одна выбранная категория;
  - категории локализованы на русский;
  - один разделитель после блока "Последние".
- Смайлы в picker масштабированы под текущий UX (`32x32`) и облачко подогнано под скролл внутри границ.
- В шапке пикера вкладки "Смайлики/GIF" увеличены до `16px`.
- В чате рендер emoji выполнен через Twemoji-картинки.
- Копирование из чата:
  - при копировании emoji преобразуются в token/unicode корректно, без визуальных артефактов выделения.
- Ввод emoji:
  - поле ввода переведено на редакторную модель (`EditorContent`/Tiptap) вместо `textarea overlay`.
  - вставка из picker и paste через `:shortcode:` обрабатываются как emoji-ноды.
  - при отправке emoji-ноды конвертируются в `:emoji_name:` для стабильного хранения/рендера.
- Размеры emoji в сообщениях:
  - `1 = 48px`, `2-5 = 32px`, `>5 = 24px`, `emoji + текст = 24px`.

## GIF система в чате (актуально)

- Источник GIF: серверный endpoint `/tenor` (прокси к Tenor), лимит выдачи `10`.
- В пикере GIF:
  - без поиска показываются "последние использованные GIF" (до `10`) + добивка из Tenor;
  - при клике на GIF попап закрывается и сообщение сразу отправляется в чат.
- Отправка GIF реализована как текстовое сообщение с префиксом `cmx-gif-url:`.
- На сервер не загружается бинарный файл GIF; хранится только URL.
- Рендер GIF в ленте:
  - вертикальные GIF сохраняют свою ориентацию;
  - ограничение только по высоте `max-height: 320px`;
  - ширина адаптивная (по фактическим пропорциям в пределах чата).

## Упоминания, цитаты и уведомления (актуально)

- Mention flow:
  - вставка `@имя` из контекстного меню сообщения;
  - autocomplete по `@` в редакторе;
  - доставка mention-уведомления в верхний bell-popover;
  - после открытия уведомление удаляется из локального списка mention-notifications.
- Quote flow:
  - сообщение может содержать quote token `:::cmx-quote|...:::`;
  - quote bubble в чате и quote chip в композере строятся из этого токена;
  - переход по клику работает как jump к закрепленному сообщению;
  - quote совместим с текстом, вложениями и GIF.
  - compact composer quote-chip отдельно доведен по вертикальному центрированию.

## Реакции на сообщения (актуально)

- В production-прототипе добавлены реакции в стиле Discord-подобного UI:
  - пункт `Реакции` в контекстном меню сообщения;
  - popup выбора emoji;
  - reaction-chip'ы под сообщением;
  - повторный клик по своей реакции снимает ее;
  - через кнопку `+` можно добавить другой emoji.
- Счетчик реакции показывается только начиная с `2`.
- Свои реакции мягко подсвечиваются синим.
- Реакции разрешены всем пользователям:
  - отдельный permission-gate `REACT_TO_MESSAGES` больше не требуется для toggle-flow;
  - старый UI `Protect permission={REACT_TO_MESSAGES}` тоже снят.

## Профиль текущего пользователя (актуально)

- В модалку редактирования собственного профиля добавлена смена аватарки.
- Используется существующий серверный маршрут `users.changeAvatar`.
- В settings runtime сохранение имени и аватарки теперь идет единым UX-flow.

## Группы и фильтры видимости (актуально)

- Помимо общей группы теперь есть отдельные управляемые group-channels.
- У группы есть:
  - название;
  - описание;
  - аватарка;
  - фильтр видимости (`Все`, `Сисадмины`, `Самара`).
- Видимость группы:
  - пользователи видят группу, если их фильтр совместим с фильтром группы;
  - `Все` видит все группы.
- Право управления группами:
  - наличие `MANAGE_GROUP_CHATS` обязательно;
  - если текущий пользователь не глобальный, он может создавать/редактировать/удалять только группы своего фильтра;
  - это ограничение есть и в UI, и на сервере.

## Принудительное обновление клиента после новой сборки (актуально)

- Сервер `GET /info` возвращает `buildId`.
- Клиент хранит последний `buildId` и при изменении делает однократный `reload`.
- Добавлена защита от циклического перезагрузочного сценария в рамках одной сессии.
- Это важно для локальной разработки/автореконнекта, когда вкладка не перезагружается вручную.

## Текущий handoff-статус (важно)

- `prototype-interface.tsx` находится в late-stage декомпозиции и уже в основном работает как orchestration-shell.
- Основной активный контекст сейчас не про layout, а про:
  - новую access-code модель пользователей/ролей;
  - реакции, sounds и polish production-прототипа;
  - корректный handoff между машинами через актуальный `exe` и `drizzle.zip`.
- Старый `invites.add` всё ещё существует как общий серверный механизм, но prototype settings больше не должны использовать его как основной login-flow.
- Последняя критичная исправленная проблема:
  - в `meta/_journal.json` не было записей для `0011-0013`, из-за чего на живом сервере не создавалась таблица `pending_user_activations`;
  - это уже исправлено в репозитории и должно попасть в новый `exe`.

## Ограничения по платформе и экранам (обязательно)

- Продукт разрабатывается только для desktop-использования.
- Мобильная версия (телефоны/планшеты) не планируется и не поддерживается.
- Минимальный целевой экран: `1366x768`.
- Максимальный целевой экран: `1920x1080`.
- Нужно учитывать, что окно приложения может быть немного меньше размера экрана.
- Все новые UI-решения проверять в диапазоне desktop-окон между этими границами.
